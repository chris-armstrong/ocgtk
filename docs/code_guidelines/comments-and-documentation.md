# Comments and Documentation

## Succinct Comments

- Method and type comments must be succinct. State what the code does, not the history of how it came to be.

## Keep Comments Up to Date

- When modifying code, check that adjacent comments and documentation are still accurate. Out-of-date comments are worse than no comments.

## No Ephemeral References

- Comments must not reference stages, phases, sprints, planning documents, meeting notes, or any other ephemeral artifact. Comments must remain meaningful without access to external plans or documents that may change or disappear.
